### 3. 重新梳理账号审阅执行逻辑

今天进一步确认了一个关键点：

> `是否审阅……` 类 node 不是直接执行 OCR，而是应先基于“账号审阅导出”文件判断对应层级是否存在账号，再决定是否需要 supporting / 截图 / OCR。

因此后续执行逻辑调整为：

```text
检查节点 Node
        ↓
映射到账号审阅导出中的 ReviewObject
        ↓
筛选“审阅对象”列
        ↓
提取账号名、账号使用人、用途
        ↓
过滤 SYSTEM / 系统账号
        ↓
判断是否需要手工 supporting
        ↓
如需要截图，再进入 OCR / evidence check
```

已初步确认的映射包括：

| Node        | 账号审阅导出中对应的审阅对象 |
| ----------- | -------------- |
| 需审阅发版工具发版账号 | 发布工具           |
| 需审阅发版工具后台账号 | 发布工具服务器        |
| 需审阅应用层管理员账号 | 应用层            |
| 需审阅应用层服务器账号 | 应用服务器          |
| 需审阅数据库服务器账号 | 数据库服务器         |
| 需审阅数据库账号    | 数据库            |

同时明确：

* `需上传应用层账号截图` **不是账号比对节点**；
* 它应作为 evidence completeness check；
* 后续只判断是否存在对应应用层 supporting / 截图，并关联到“需审阅应用层管理员账号”。

### 4. 明确 SYSTEM / 系统账号处理规则

确认系统账号不应进入 expected account set。

后续执行计划中会默认过滤：

* `SYSTEM`
* 系统账号
* 系统用户
* 服务账号
* service account 类账号

这些账号会被标记为 skipped，而不是进入 OCR 比对。

建议后续输出中保留：

```text
SkippedAccounts
SkipReason = SYSTEM_ACCOUNT
```

用于审计留痕。

---

### 5. 明确服务器类型逻辑来源

确认服务器类型不需要单独维护系统映射表，而是直接来自 WP 中的 `服务器类型` 字段。

后续判断服务器类节点时，应使用：

```text
Node + ReviewObject + WP服务器类型 + ReferTo / PaaS / PAM
```

初步规则方向：

| 场景              | 处理思路                     |
| --------------- | ------------------------ |
| 应用层 / 发布工具前台    | 通常需要手工 supporting        |
| 数据库账号           | 通常基于账号审阅导出，不强制截图         |
| Linux 服务器       | 一般优先看系统导出                |
| Windows 服务器     | 需要进一步检查用户组内成员 supporting |
| DB PaaS / Refer | 跳转公共系统或对应 TargetSystem   |
| SOD / 开发人员生产权限  | 后续做派生集合判断                |

## 二、今日关键发现

1. 原先的 Scope Engine 只解决“哪些 node 进入检查范围”，但不解决“具体怎么查”。
2. OCR 不能直接作为下一步，需要先生成 expected account set。
3. `账号审阅导出` 是连接 scope 与 evidence check 的关键中间层。
4. 部分截图不是普通账号表，而是角色 / 权限矩阵，例如 Jenkins 打勾矩阵，不能简单按账号清单截图处理。
5. `需上传应用层账号截图` 不是账号核对，而是材料存在性检查。
6. SYSTEM 等系统账号应默认过滤，不进入人工截图核对范围。

## 四、下一步计划

下一步计划推进 `Execution Plan Builder`，目标是生成一个三 sheet 的执行计划文件：

### 1. `expected_accounts`

从账号审阅导出中筛选 expected accounts。

包括：

* TaskID
* System
* Node
* ReviewObject
* Account
* AccountOwner
* Usage
* IsSystemAccount
* ExpectedStatus
* SkipReason

### 2. `evidence_plan`

记录该 node 是否需要 supporting / 截图 / OCR。

包括：

* TaskID
* System
* Node
* ReviewObject
* ServerType
* NeedManualEvidence
* EvidenceFormat
* ParseMethod
* EvidenceStatus
* Remark

### 3. `compare_result`

作为后续 OCR / Excel 读取后的比对结果表。

包括：

* ExpectedAccounts
* ActualAccounts
* MissingAccounts
* ExtraAccounts
* UsageMatchStatus
* CheckResult
* Remark
